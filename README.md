# BNR info-hub

This repository contains the development files for the BNR `info-hub`.

The `info-hub` is being developed as part of BNR Refit Phase 2. Its purpose is to provide a structured digital home for BNR outputs, documentation, operational guidance, and reproducible reporting workflows.

The site is intended to support both public-facing and internal-facing needs. It will help the BNR publish clear surveillance outputs, maintain a documented Operations Manual, organise technical guidance, and support a more reproducible approach to routine reporting.

The `info-hub` is not just a website. It is part of a wider registry refit designed to strengthen how BNR data are prepared, analysed, documented, reviewed, and published.

## Site purpose

The purpose of the `info-hub` site is to provide:

- a clear public entry point for BNR surveillance outputs
- a home for routine reports, dashboards, slide decks, and summary products
- a structured Operations Manual for registry processes and handover
- technical documentation for the analytics and publication workflow
- a stable publication layer for outputs generated by Stata and supporting scripts
- a simple way to separate generated analytical outputs from manually written interpretation
- a future-ready structure that can support cardiovascular disease, primary care surveillance, and later registry extensions

The site is designed to support routine publication while keeping the underlying compute workflow separate and reproducible.

The basic model is:

```text
Stata scripts -> generated outputs -> Quarto publication site
```

This means that the site should publish and explain outputs, but the main analytics should remain in documented Stata scripts.

## Background

The Barbados National Registry is moving toward a more structured and reproducible surveillance and reporting model.

BNR Refit Phase 2 focuses on improving the architecture around registry analytics and reporting. This includes:

- clearer data governance and release processes
- reproducible Stata-based analytics
- standardised outputs for tables, figures, and reporting datasets
- a Quarto-based publication framework
- an Operations Manual to support continuity and handover
- a pathway for low-code execution by future users
- future expansion into primary care surveillance, including hypertension and diabetes

The `info-hub` supports this work by giving the project a single organised place for website content, reporting products, documentation, and publication infrastructure.

## Repository purpose

The purpose of this repository is to support:

- local development of the BNR Quarto website
- publication of reports, dashboards, slides, and operational documentation
- separation between analytical computation and publication
- reproducible generation of tables, figures, logs, and build metadata
- future handover to users with strong Stata skills but limited Git, Python, or Quarto experience

## Main tools

The main tools used in this project are:

- **Stata** — primary analytics and compute tool
- **Quarto** — website, report, PDF, slide, and dashboard rendering
- **Git and GitHub** — version control and project collaboration
- **Python** — lightweight support utilities only
- **PowerShell** — local helper scripts and setup commands on Windows

Stata remains the main compute engine. Python is used only where it provides simple support for file handling, visualisation, metadata, or Quarto dashboard experiments.

## Folder summary

### `scripts/`

Stores Stata code and small support scripts.

Expected subfolders:

```text
scripts/
├── stata/
├── python/
└── powershell/
```

Use `scripts/stata/` for the main analytics workflow.

Use `scripts/python/` only for lightweight support utilities.

Use `scripts/powershell/` for simple local helper scripts.

### `outputs/`

Stores machine-generated outputs.

Expected subfolders:

```text
outputs/
├── tables/
├── figures/
├── data/
├── logs/
└── build/
```

Files in `outputs/` should normally be generated by scripts and should not usually be manually edited.

### `site/`

Stores the Quarto website source files.

Expected subfolders:

```text
site/
├── reports/
├── slides/
├── dashboards/
├── operations-manual/
├── technical/
└── assets/
```

The Operations Manual is treated as a normal section of the Quarto website, not as a separate Quarto book.

### `docs/`

Stores setup, developer, and handover documentation that may not be part of the public-facing website.

### `assets/`

Stores shared static project assets such as logos, diagrams, and source image files.

### `setup-checks/`

Stores lightweight setup validation files.

### `.github/`

Stores GitHub-specific configuration, including future GitHub Actions workflows.

## Output contract

The main workflow is:

```text
Stata scripts -> outputs folder -> Quarto site
```

This means:

- scripts create outputs
- outputs store generated files
- Quarto pages publish, describe, or link to those outputs

The Quarto site should not contain hidden analytics logic that replaces the Stata compute workflow.

## Naming rules

Use dashes for human-facing website files.

Examples:

```text
annual-cvd-report.qmd
bnr-refit-overview.qmd
cvd-dashboard.qmd
```

Use underscores for machine-generated analytic outputs.

Examples:

```text
table_cvd_incidence_overall_2024_national.csv
figure_cvd_incidence_overall_2019_2024_national.png
build_metadata_20260417_1015.yml
```

## Local setup

Local setup is documented in the setup guide.

Start with:

```text
docs/setup/
```

Each user should maintain their own local setup notes in:

```text
local-setup-notes.md
```

This file is ignored by Git because it may contain machine-specific paths, usernames, and troubleshooting notes.

## Python dependencies

The core Python package list is stored in:

```text
requirements.txt
```

Install the packages with:

```powershell
python -m pip install -r requirements.txt
```

The exact package versions installed on a machine can be recorded with:

```powershell
python -m pip freeze > requirements-freeze.txt
```

## Development principles

This project follows these principles:

- keep the structure simple
- keep Stata as the main compute tool
- keep Python lightweight and secondary
- separate compute from publication
- avoid hidden manual changes to generated outputs
- document decisions and changes as they happen
- make the system understandable to future users
