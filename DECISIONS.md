# Architectural Decisions

This file records major architectural decisions for the BNR `info-hub`.

Its purpose is to:

- prevent architectural drift
- document why important choices were made
- support handover to future users
- preserve institutional memory
- make future changes deliberate rather than accidental

This is not a task list. It is a decision register.

Minor edits do not need to be recorded here. Structural, methodological, workflow, or tooling decisions should be recorded.

---

# Decision Register

---

## AD-001: Stata is the primary analytics engine

**Decision**

Stata is the primary analytics and compute tool for the BNR `info-hub` system.

**Reason**

The project is designed for users with strong Stata skills. Keeping Stata as the main compute tool supports continuity, readability, handover, and operational sustainability.

**Implications**

- Core analytics should be written in Stata.
- Stata scripts should generate tables, figures, data extracts, logs, and build metadata.
- Python should not replace Stata as the main analytics engine unless a future decision explicitly changes this.

---

## AD-002: Python is a lightweight support tool

**Decision**

Python may be used only for lightweight support tasks.

Permitted uses include:

- file handling
- YAML or metadata support
- simple visualisation support
- Quarto dashboard experiments
- small utility scripts

**Reason**

Python is useful, but introducing a large Python dependency would increase handover risk for a Stata-first team.

**Implications**

- Python dependencies should remain minimal.
- New Python packages should only be added when there is a clear project need.
- Python code should be placed in `scripts/python/`.
- Core surveillance analytics should remain in Stata.

---

## AD-003: Quarto is the publication framework

**Decision**

Quarto is used for the BNR `info-hub` publication layer.

It supports:

- website pages
- reports
- PDF outputs
- RevealJS slides
- dashboards
- technical documentation
- the Operations Manual

**Reason**

Quarto provides a flexible and reproducible publication system while keeping the project static and relatively simple.

**Implications**

- Quarto source files are stored under `site/`.
- Website pages, reports, slides, dashboards, and documentation can share one publication framework.
- The site should publish or explain outputs created by the compute layer rather than replacing the compute layer.

---

## AD-004: The published site is static

**Decision**

The published `info-hub` site is a static site.

**Reason**

A static site is easier to host, easier to hand over, and avoids server maintenance. It also reduces operational risk.

**Implications**

- No live database connection is required for the public site.
- No server-side application is required for routine publication.
- Dashboards should be static or pre-rendered.
- Any interactivity should be browser-side and should not depend on a live backend.

---

## AD-005: Compute and publication layers are separated

**Decision**

The compute layer and publication layer are kept separate.

The basic model is:

`Stata scripts -> generated outputs -> Quarto publication site`

**Reason**

This separation makes the system easier to audit, troubleshoot, and hand over.

**Implications**

- Stata scripts create outputs.
- Outputs are stored in `outputs/`.
- Quarto pages read from, link to, or describe generated outputs.
- Quarto pages should not contain hidden analytics logic that replaces the Stata workflow.

---

## AD-006: The `outputs/` folder is the contract between compute and publication

**Decision**

The `outputs/` folder stores machine-generated outputs used by the publication layer.

The expected structure is:

- `outputs/tables/`
- `outputs/figures/`
- `outputs/data/`
- `outputs/logs/`
- `outputs/build/`

**Reason**

A stable outputs structure creates a clear handover point between scripts and publication files.

**Implications**

- Files in `outputs/` should normally be generated by code.
- Generated outputs should not usually be manually edited.
- If an output is wrong, the preferred fix is to update the script that created it and regenerate the output.
- Logs and build metadata form part of the audit trail.

---

## AD-007: Machine-generated outputs use underscores

**Decision**

Machine-generated output filenames use underscores.

Example:

`table_cvd_incidence_overall_2024_national.csv`

**Reason**

Underscores are safer for filenames that may later be referenced by Stata, Python, metadata files, or build scripts.

**Implications**

- Generated files in `outputs/` should use underscores.
- Names should be lowercase, readable, sortable, and stable.
- The broad naming pattern is:

`product_domain_metric_population_period_granularity_version.ext`

Not every filename needs every component, but the order should remain consistent where practical.

---

## AD-008: Human-facing website files use dashes

**Decision**

Human-facing website files use dashes.

Examples:

- `annual-cvd-report.qmd`
- `bnr-refit-overview.qmd`
- `cvd-dashboard.qmd`

**Reason**

Dashes are readable in links, URLs, and website navigation.

**Implications**

- Quarto pages, report files, slide files, and dashboard files should generally use dashes.
- This rule applies mainly inside `site/`.
- This complements the underscore rule for machine-generated outputs.

---

## AD-009: The Operations Manual is part of the wider Quarto site

**Decision**

The Operations Manual is treated as a normal section of the Quarto website, not as a separate Quarto book.

**Reason**

A separate Quarto book would add complexity. The project prioritises simplicity and handover readiness.

**Implications**

- Operations Manual files are stored in `site/operations-manual/`.
- The manual can still use structured navigation.
- The project avoids maintaining a separate book-within-site structure unless a future decision changes this.

---

## AD-010: Reports, slides, and dashboards are separated within the site

**Decision**

Different Quarto publication types are stored in separate folders.

Expected folders include:

- `site/reports/`
- `site/slides/`
- `site/dashboards/`
- `site/operations-manual/`
- `site/technical/`
- `site/assets/`

**Reason**

Reports, slides, dashboards, and manual pages behave differently. Separating them makes the site easier to understand and maintain.

**Implications**

- Report-style pages belong in `site/reports/`.
- RevealJS slide decks belong in `site/slides/`.
- Dashboard pages belong in `site/dashboards/`.
- Operations Manual content belongs in `site/operations-manual/`.
- Technical site documentation belongs in `site/technical/`.

---

## AD-011: Local setup notes are machine-specific

**Decision**

Each user may maintain a `local-setup-notes.md` file, but it is ignored by Git.

**Reason**

Local setup notes may contain usernames, paths, machine details, troubleshooting notes, and local decisions.

**Implications**

- `local-setup-notes.md` is listed in `.gitignore`.
- A safe template may be created later if useful.
- Users should not rely on another person’s local setup notes as project documentation.

---

## AD-012: `.gitkeep` files may be used to preserve empty folders

**Decision**

`.gitkeep` files may be used to make empty folders visible in GitHub.

**Reason**

Git tracks files, not empty folders. `.gitkeep` allows the intended scaffold to be visible before real files are added.

**Implications**

- `.gitkeep` files are harmless.
- They may remain in folders after real files are added.
- They may also be deleted later if they become visually distracting.

---

## AD-013: `.gitignore` protects generated and local files

**Decision**

The repository uses `.gitignore` to prevent local clutter, generated files, cache files, and machine-specific notes from being committed.

**Reason**

This keeps the repository clean and reduces the risk of committing local or generated files unintentionally.

**Implications**

- Python caches and environments are ignored.
- Quarto caches and rendered site output are ignored.
- Generated outputs are ignored during setup, while `.gitkeep` placeholders remain trackable.
- `local-setup-notes.md` is ignored.

---

## AD-014: Documentation is part of the system

**Decision**

Documentation is treated as a core part of the BNR `info-hub`, not as an optional extra.

**Reason**

The project must be understandable to future users, especially Stata users who may be new to Git, Python, Quarto, and website development.

**Implications**

- Setup documentation should be step-by-step.
- Major decisions should be recorded in `DECISIONS.md`.
- Meaningful workflow or methodological changes should be recorded in `CHANGELOG.md`.
- The Operations Manual should evolve alongside the system.

---

## AD-015: Simplicity is preferred over sophistication

**Decision**

When several technical options are possible, the simpler and more handover-ready option should be chosen.

**Reason**

The system needs to remain usable by future staff and collaborators with mixed technical backgrounds.

**Implications**

- Avoid unnecessary dependencies.
- Avoid unnecessary automation layers.
- Avoid complex framework choices unless clearly justified.
- Prefer readable Stata and clear documentation over clever code.
- Any new pathway should consider whether the Operations Manual also needs updating.

---

# Decision Change Process

If a future change affects the project architecture, workflow, tooling, or handover model:

1. Add a new decision entry.
2. Describe the decision.
3. Explain why the change was needed.
4. Record the implications.
5. Update related documentation if needed.

No major architectural shift should happen silently.

---

# Guiding Principle

Architecture is governance.

Every structural decision affects how the BNR `info-hub` will be maintained, audited, and handed over.
